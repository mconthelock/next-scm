# I18N Guide

เอกสารนี้อธิบายโครงสร้างรองรับ 2 ภาษาในโปรเจกต์นี้ โดยตอนนี้รองรับภาษาไทย (`th`) และภาษาอังกฤษ (`en`)

## เป้าหมายของโครงสร้างนี้

โครงสร้างที่เพิ่มเข้ามามีเป้าหมาย 4 อย่าง

1. ให้ระบบจำภาษาที่ผู้ใช้เลือกได้
2. ให้ทั้งหน้า public, auth และหน้าภายในใช้แหล่งข้อความกลางเดียวกัน
3. ให้เปลี่ยนภาษาได้โดยไม่ต้องแยก route เป็น `/th/...` และ `/en/...` ในรอบนี้
4. ให้ขยายต่อไปสู่การแปลข้อความจาก API หรือ database ได้ง่าย

## ภาพรวมการทำงาน

ลำดับการทำงานของระบบภาษาตอนนี้เป็นแบบนี้

1. `app/layout.tsx` อ่าน cookie ชื่อ `scm-locale`
2. ถ้า cookie มีค่า `th` หรือ `en` ระบบจะใช้ค่านั้น
3. ถ้าไม่มี cookie ระบบจะ fallback ไปที่ `th`
4. layout ส่งค่า `initialLocale` เข้า `LocaleProvider`
5. component ฝั่ง client เรียก `useLocale()` เพื่ออ่าน `locale`, `messages` และ `setLocale()`
6. เมื่อผู้ใช้กด `LocaleSwitcher` ระบบจะอัปเดต state, cookie และ `document.documentElement.lang`

ไฟล์นี้มีส่วนหลักดังนี้

1. `config.ts`
   ความหมาย: เก็บ `Locale`, cookie name, default locale และ helper สำหรับ resolve locale

2. `types.ts`
   ความหมาย: เก็บ TypeScript interfaces ของข้อความแต่ละหมวด

3. `messages/*`
   ความหมาย: แยกข้อความตาม feature เช่น auth, header, footer ทำให้เพิ่มหน้าใหม่แล้วไม่ต้องแก้ไฟล์รวมก้อนใหญ่

4. `lib/i18n.ts`
   ความหมาย: รวม export และประกอบ `messagesByLocale` กลับมาเป็น API กลางให้ไฟล์อื่นใช้งานเหมือนเดิม

5. `translateMenuLabel()`
   ความหมาย: helper สำหรับแปลชื่อเมนูแบบง่าย ๆ โดยอิงจาก label ที่มีอยู่

## LocaleProvider ทำอะไร

[components/providers/LocaleProvider.tsx](/Users/onthelock/my/php/src/next-scm/components/providers/LocaleProvider.tsx) ทำหน้าที่เป็น state กลางของภาษาในฝั่ง client

สิ่งที่ provider นี้เก็บไว้คือ

- `locale`: ภาษาปัจจุบัน
- `messages`: ชุดข้อความของภาษาปัจจุบัน
- `setLocale()`: ฟังก์ชันสำหรับสลับภาษา

เมื่อเรียก `setLocale()` จะเกิดสิ่งต่อไปนี้

1. state เปลี่ยน
2. cookie `scm-locale` ถูกอัปเดต
3. ค่า `lang` บน `<html>` ถูกอัปเดตฝั่ง client

## เมนูจาก API ทำงานอย่างไรกับ i18n

เมนูของระบบยังถูกดึงจาก `/api/menu/me` เหมือนเดิม โดย route นี้จะอ่าน `groupId` จาก session ฝั่ง server ก่อนแล้วค่อยไปดึงเมนูจริงจาก mock API

เรื่องภาษาในเมนูตอนนี้ใช้วิธีง่ายก่อน คือ

1. ดึงเมนูเดิมจาก API ตามปกติ
2. ตอน render ใน [components/layout/Header.tsx](/Users/onthelock/my/php/src/next-scm/components/layout/Header.tsx) จะเรียก `translateMenuLabel()`
3. ถ้ามีคำแปลใน `messages.menuLabels` ก็ใช้คำแปล
4. ถ้าไม่มีคำแปล จะใช้ข้อความเดิมจาก API

แนวทางนี้เหมาะกับช่วงเริ่มต้น เพราะไม่ต้องเปลี่ยน schema ของ backend ทันที

ถ้าต้องการให้สมบูรณ์กว่านี้ในอนาคต มี 2 ทางที่แนะนำ

1. เปลี่ยน backend/menu config ให้ส่ง `titleKey` กลับมาแทนข้อความตรง ๆ
2. หรือส่งทั้ง `titleTh` และ `titleEn` กลับมาจาก API

## วิธีเพิ่มข้อความใหม่

ถ้าต้องการเพิ่มข้อความใหม่ ให้ทำตามลำดับนี้

1. เลือกหมวดข้อความก่อน เช่นถ้าเป็นหน้า login ให้แก้ที่ [lib/i18n/messages/auth.ts](/Users/onthelock/my/php/src/next-scm/lib/i18n/messages/auth.ts)
2. ถ้าต้องเพิ่ม field ใหม่ ให้เพิ่ม type ใน [lib/i18n/types.ts](/Users/onthelock/my/php/src/next-scm/lib/i18n/types.ts)
3. เพิ่มค่าภาษาไทยและอังกฤษในไฟล์ message ของหมวดนั้น
4. ไปยัง component ที่ต้องการ แล้วเรียก `const { messages } = useLocale()`
5. ใช้ข้อความจาก `messages` แทน hardcoded string

ตัวอย่างแนวคิด

```tsx
const { messages } = useLocale();

<button>{messages.header.logout}</button>;
```

## วิธีเพิ่มภาษาที่ 3

ถ้าต้องการเพิ่มภาษาใหม่ เช่นญี่ปุ่น (`ja`) ให้ทำตามนี้

1. เพิ่ม `'ja'` ใน type `Locale`
2. เพิ่ม `'ja'` ใน `SUPPORTED_LOCALES`
3. เพิ่มค่า `ja` ในแต่ละไฟล์ใต้ [lib/i18n/messages](/Users/onthelock/my/php/src/next-scm/lib/i18n/messages)
4. อัปเดต `LocaleSwitcher` ให้มีปุ่มภาษาใหม่

## ข้อจำกัดของโครงสร้างปัจจุบัน

ตอนนี้ระบบรองรับสองภาษาได้ในระดับ UI แล้ว แต่ยังมีข้อจำกัดที่ควรรู้

1. ยังไม่ได้ใช้ localized route เช่น `/th/login` หรือ `/en/login`
2. ข้อความจาก backend หรือ database ที่ส่งมาเป็น string ตรง ๆ ยังไม่ได้แปลอัตโนมัติทั้งหมด
3. metadata ของแต่ละหน้ายังไม่ได้แยกตามภาษา
4. ยังไม่มี test สำหรับตรวจว่าทุก key มีครบทั้งสองภาษา

## แนวทางขยายต่อที่แนะนำ

ถ้าจะพัฒนาต่อให้ระบบ i18n แข็งแรงขึ้น แนะนำลำดับนี้

1. แยก key ของข้อความใน menu ให้มาจาก backend เป็น `titleKey`
2. เพิ่ม helper สำหรับตรวจ missing translation key ตอน build หรือ lint
3. แยก metadata ตาม locale ในแต่ละหน้า
4. ถ้าต้องการ SEO หลายภาษา ค่อยพิจารณาเปลี่ยนไปใช้ locale segment ใน route

## สรุปสั้นที่สุด

โครงสร้าง i18n ตอนนี้ใช้ `cookie + LocaleProvider + messages/* แยกตาม feature` เป็นแกนกลาง ทำให้แอปสลับภาษาไทย/อังกฤษได้จาก UI, จำค่าภาษาได้, และเพิ่มข้อความของหน้าใหม่โดยไม่ต้องเอาทุกอย่างมากองไว้ในไฟล์เดียว
